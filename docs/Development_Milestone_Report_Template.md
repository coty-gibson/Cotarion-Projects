# Development Milestone Report

Use this report at the completion of each operational capability. Omit routine implementation detail that does not help the Product Owner operate, evaluate, or govern Cotarion.

## 1. Capability Completed

Briefly identify the completed business capability and the business outcome it provides.

## 2. Operational Workflow

Describe what the Product Owner can now accomplish in the platform from beginning to end.

Include:

- Entry point
- Primary workflow
- Saved or generated business records
- Relevant downstream connections
- How to exercise the capability

## 3. Technical Summary

Summarize material engineering work:

- Major implementation work
- Significant refactoring
- Test coverage additions or changes
- Performance improvements
- Security improvements
- Validation results

Do not enumerate routine file edits unless they materially explain the capability.

## 4. Architecture Notes

Record:

- New ADRs
- Significant architectural decisions
- Architectural assumptions requiring future review

If none occurred, state that the capability followed approved architecture without introducing a new architectural decision.

## 5. Critical Defects

For every Critical Defect, record:

- Defect
- Workflow impact
- Root cause
- Resolution status
- Validation result
- Remaining risk

If none were found, state that no Critical Defects remain.

## 6. Product Owner Attention Required

Include only decisions requiring Product Owner authority:

- Conflicting business rules
- Business methodology decisions
- Naming requiring business approval
- Product policy questions
- Security or compliance decisions
- An operational workflow that cannot reasonably proceed without Product Owner direction

If no decision is required, state **None**. Do not use this section for routine engineering choices.

## 7. Product Owner Refinement Backlog

List Workflow Improvements observed during implementation or validation.

For each observation:

- Reference its permanent backlog ID
- Summarize the operational observation
- State its current lifecycle status

Recording an observation does not interrupt the next capability.

## 8. Next Capability

Name the next approved operational capability Development will begin.

If the next capability is blocked by an approved escalation condition, identify the condition and the Product Owner decision required.

## Milestone Evidence

Attach or link only evidence useful for operational validation:

- Running application URL
- Login instructions
- Screen map
- Workflow instructions
- Screenshots
- Test and build results
- Known operational limitations

## Reporting Standard

A milestone report is complete when the Product Owner can understand:

1. What Cotarion can now do.
2. How to exercise it.
3. Whether it is technically and operationally validated.
4. Whether Product Owner authority is required.
5. What capability Development proceeds to next.
